# Internationalization (i18n)

gemi's i18n is built around **component-scoped dictionaries**. Instead of one giant global namespace, each component (page, panel, form, email) gets its own `Dictionary` of keyed strings, and gemi prefetches only the dictionaries a route actually needs. Locale detection, translation, and the client hooks all build on these dictionaries.

## Dictionaries

Create a dictionary with `Dictionary.create` from `gemi/i18n`. Each entry is a key mapping to a per-locale string:

```typescript
// app/i18n/HomePage.ts
import { Dictionary } from "gemi/i18n";

export const HomePage = Dictionary.create("HomePage", {
  title: {
    "en-US": "Welcome to Gemi {{version}}",
    "tr-TR": "Gemi'ye Hoş Geldiniz {{version}}",
  },
  description: {
    "en-US": "A simple and fast framework for building web applications.",
    "tr-TR": "Web uygulamaları oluşturmak için basit ve hızlı bir çerçeve.",
  },
});
```

`Dictionary.create(name, translations)`:

- `name` — a unique dictionary name (matches the component it serves). It becomes the key you pass to `useTranslator` on the client.
- `translations` — `{ [key]: { [locale]: string } }`. Every key holds a map from locale code (e.g. `"en-US"`) to the translated string.

### Interpolation

Strings support `{{param}}` placeholders, filled in at render time:

```typescript
title: {
  "en-US": "Welcome to Gemi {{version}}",
}
```

```tsx
t("title", { version: "1.0" }); // "Welcome to Gemi 1.0"
```

Placeholders also support **type casting** with `{{param:type}}` (`number`, `string`, `boolean`):

```typescript
"en-US": "You have {{count:number}} items";
```

And **interpolated content** with `{{param:[content]}}`, where the param is a function that receives the bracketed content — this is how you embed JSX (links, bold text) inside a translation:

```tsx
// "Read our {{terms:[terms of service]}} to continue"
t.jsx("notice", {
  terms: (label) => <a href="/terms">{label}</a>,
});
```

> **Note:** gemi has no dedicated pluralization API. Handle plurals with a `{{count:number}}` placeholder plus wording, a function param, or separate keys per plural form.

## Collecting dictionaries

Gather every dictionary in `app/i18n/index.ts` and export them as a single object:

```typescript
// app/i18n/index.ts
import { HomePage } from "./HomePage";
import { About } from "./About";
import { Footer } from "./Footer";
import { Header } from "./Header";

export default { HomePage, About, Footer, Header };
```

Small apps can define the dictionaries inline in `index.ts`; larger apps (like production `folio`, with 7 locales — en/tr/de/it/fr/es/ru) keep one file per component (`app/i18n/<Component>.ts`) and collect them here.

## Prefetching per route

Register your dictionaries and locale config in `I18nServiceProvider`. This is where you tell gemi which dictionaries each route needs so they can be sent with the initial render:

```typescript
// app/kernel/providers/I18nServiceProvider.ts
import { I18nServiceProvider } from "gemi/i18n";
import type { HttpRequest } from "gemi/http";
import components from "@/app/i18n";

export default class extends I18nServiceProvider {
  defaultLocale = "en-US";
  supportedLocales = ["en-US", "tr-TR"];

  prefetch = {
    "/": [components.HomePage],
    "/about": [components.About],
  };

  detectLocale(req: HttpRequest) {
    return req.cookies.get("i18n-locale") ?? null;
  }

  onLocaleChange(locale: string) {
    console.log(`Locale changed to ${locale}`);
  }
}
```

Configuration fields:

- **`defaultLocale`** — fallback locale (default `"en-US"`).
- **`supportedLocales`** — `string[]` of accepted locale codes. A requested locale outside this list falls back to `defaultLocale`.
- **`prefetch`** — `Record<routePattern, Dictionary[]>`. Maps each route pattern to the dictionaries needed to render it; only these are shipped for that route. A route commonly lists shared dictionaries (`Header`, `Footer`) plus its own.
- **`detectLocale(req)`** — resolve the locale for a request. Return a locale string or `null` to fall back. A typical implementation reads the `i18n-locale` cookie, then the `x-locale`/`accept-language` headers.
- **`onLocaleChange(locale)`** — hook called when the active locale changes.

## Client usage

### `useTranslator`

`useTranslator(componentName)` returns a translator `t` bound to that component's dictionary and the current locale:

```tsx
import { useTranslator } from "gemi/client";

function HomePage() {
  const t = useTranslator("HomePage");

  return (
    <div>
      <h1>{t("title", { version: "1.0" })}</h1>
      <p>{t("description")}</p>
    </div>
  );
}
```

- `t(key, params?)` — returns the translated string for `key` in the current locale, with `params` interpolated. Params are required only when the string has placeholders.
- `t.jsx(key, params?)` — same, but typed to return a JSX element (use it when a param renders JSX, e.g. an embedded link).

If a translation can't be resolved, `t` logs a warning and returns the key itself, so the UI degrades gracefully rather than crashing.

### `useLocale`

`useLocale()` returns the current locale and a setter:

```tsx
import { useLocale } from "gemi/client";

function LocaleSwitcher() {
  const [locale, setLocale] = useLocale();

  return (
    <select value={locale} onChange={(e) => setLocale(e.target.value)}>
      <option value="en-US">English</option>
      <option value="tr-TR">Türkçe</option>
    </select>
  );
}
```

`setLocale(locale)` persists the choice in the `i18n-locale` cookie and re-navigates so the new locale's dictionaries load.

## Server-side translation

### `I18n` facade

Use the [`I18n`](./facades.md) facade to read the request's locale on the server:

```typescript
import { I18n } from "gemi/facades";

const locale = I18n.locale(); // active locale for this request
I18n.getDefaultLocale();
I18n.getSupportedLocales();
```

### `Dictionary.text` and `Dictionary` render helpers

For strings that live on the server (validation messages, emails, generated content), translate inline without a client dictionary:

```typescript
import { Dictionary } from "gemi/i18n";

// Inline locale map -> string in the current (or given) locale
const message = Dictionary.text({
  "en-US": "Slot name is required",
  "tr-TR": "Slot adı gereklidir",
});
```

`Dictionary.text(content, args?)` picks `content[locale]` (defaulting to `I18n.locale()`), interpolating `args.params`. This is common in `HttpRequest` validation schemas:

```typescript
class CreateSlotRequest extends HttpRequest<CreateSlotInput> {
  schema = {
    name: {
      required: Dictionary.text({
        "en-US": "Slot name is required",
        "tr-TR": "Slot adı gereklidir",
      }),
    },
  };
}
```

An instance dictionary also offers **server-only** helpers:

- `dictionary.render(key, { locale?, params? })` — resolve a single key to a string in the given (or current) locale, with interpolation.
- `dictionary.reference(key)` — return the full per-locale map for a key.

> **Note:** `render` and `reference` throw if called in the browser — they are server-only. On the client, use `useTranslator`.

## Locale in emails

Emails render outside the browser, so pass the resolved locale into the email component explicitly and use it to select strings. `Dictionary.text` works here too — pass the target `locale` in its args (it otherwise defaults to `I18n.locale()`):

```tsx
import { I18n } from "gemi/facades";
import { Dictionary } from "gemi/i18n";

// When dispatching the email
sendPinEmail({ locale: I18n.locale(), pin });

// Inside the email component
function PinEmail({ locale, pin }: { locale: string; pin: string }) {
  const heading = Dictionary.text(
    { "en-US": "Your login code", "tr-TR": "Giriş kodunuz" },
    { locale },
  );
  return <Layout locale={locale}>{heading}: {pin}</Layout>;
}
```
